# Issue #10 — Exit Codes + `--connect-timeout` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans.

**Goal:** Map typed errors to stable exit codes (empty results stay exit 0), add a `--connect-timeout` connect deadline via Zenoh's own config, and fix `info`'s mode misclassification.

**Architecture:** `ZemonError::exit_code()` gives per-kind codes; `main` exits with it. `--connect-timeout` sets `connect/timeout_ms` for client mode in `to_zenoh_config` (client already `exit_on_failure=true`), leaving peer (0 routers = healthy) untouched. `session_info` takes the configured `ConnectMode` instead of guessing from router presence, and reports `connected`.

## Global Constraints

- Empty successful query → exit 0 (`{"count":0,"items":[]}` from #7), never an error code.
- Exit codes: `internal=1`, `invalid_input=2`, `connection=3`, `timeout=4`, `not_found=5`. `0`=success.
- `--connect-timeout` is a duration string (reuses `parse_duration_arg`).
- Only user-facing connect deadline; core admin/discover timeouts unchanged.

---

### Task 1: `exit_code()` + tests + wire into main
**Files:** `crates/zemon-core/src/error.rs`, `crates/zemon-cli/src/main.rs`.
- `impl ZemonError { pub fn exit_code(&self) -> i32 { match kind ... } }`.
- `main`: `std::process::exit(e.exit_code())` instead of `exit(1)`.
- Tests: each kind → expected code.
- Commit.

### Task 2: `--connect-timeout` + config wiring
**Files:** `crates/zemon-core/src/config.rs`, `crates/zemon-cli/src/cli.rs`, `crates/zemon-cli/src/main.rs`.
- `ZemonConfig.connect_timeout: Option<Duration>`; default None; `from_env` reads `ZEMON_CONNECT_TIMEOUT` (humantime).
- `to_zenoh_config`: when `connect_timeout` set and `mode == Client`, insert `connect/timeout_ms` = ms.
- CLI: global `--connect-timeout <dur>` (value_parser `parse_duration_arg`); `build_config` sets it.
- Tests: with connect_timeout+client, `to_zenoh_config().get_json("connect/timeout_ms")` reflects the ms; with peer it stays default; builds Ok.
- Commit.

### Task 3: fix `info` mode + connected
**Files:** `crates/zemon-core/src/info.rs`, `crates/zemon-core/src/types.rs`, `crates/zemon-cli/src/main.rs`.
- `SessionDetail` gains `connected: bool`.
- `session_info(session, mode: ConnectMode)` sets `mode` from the configured mode; `connected = !routers.is_empty() || !peers.is_empty()`.
- Update Info arm: pass `config.mode`; human output prints `Connected: yes/no`.
- Build + `cargo test`.
- Commit.

## Self-Review

- Exit codes stable and consistent with #6 JSON `kind`. Empty query already exits 0 (no error raised).
- `--connect-timeout` uses Zenoh's connect deadline for client; peer's 0-router health preserved. Live behavior is manual; config wiring is unit-tested via `get_json`.
- `info` reports the real configured mode + connectivity, fixing the router-presence guess.
