# Issue #8 — `pub --json` Output Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans.

**Goal:** In `--json` mode, `pub` writes one action-result object to stdout and nothing duplicated to stderr; non-JSON mode keeps its existing stderr UX.

**Architecture:** A `zemon_core::output::publish_accepted_json` helper renders `{"ok":true,"status":"accepted","key_expr":...,"bytes":N}` (+ optional `attachment_bytes`). The `Pub` arm branches on `cli.json`.

## Global Constraints

- `status` is `"accepted"` (local stack accepted the put; not a delivery ACK) — documented.
- `bytes = value.as_bytes().len()`; `attachment_bytes = att.as_bytes().len()` when present, omitted otherwise.
- Action result does NOT use the #7 `{count,items}` collection envelope.
- `put` failure → structured error (#6).

---

### Task 1: `publish_accepted_json` helper + tests
- Add to `crates/zemon-core/src/output.rs`.
- Tests: without attachment (exact string matches spec example), with attachment (`attachment_bytes` present).
- Commit.

### Task 2: Wire into `Pub` arm
- On `cli.json`: print the helper output to stdout; skip the stderr message.
- Else: keep existing `eprintln!` UX.
- Build + `cargo test`. Commit.
